🏛️ Patrones arquitectónicos detectados

A. Multi-tenancy doble

pos_companies (legacy original) + volvix_tenants (modelo SaaS nuevo) coexisten. Las tablas hijas (productos, ventas) usan tenant_id uuid para aislar.

B. SSOT en giros_maestro.metadata (JSONB)

295 giros con todo dentro de metadata: terminologías, módulos enabled, productos plantilla, cadena de valor, competidores. Permite cambios sin DDL.

C. Tracking en vivo

volvix_giro_searches append-only para auditoria. volvix_visitor_presence upsert por session_id con TTL 24h vía DELETE on-demand (5% probability).

D. RLS (Row Level Security)

Mayoría de tablas tienen RLS habilitado: service_role full access, authenticated por tenant_id = auth.jwt() ->> 'tenant_id', anon solo SELECT en tablas marketplace públicas.

E. Multi-divisa

pos_sales y pos_products referencian currencies.code + guardan fx_rate_to_base para snapshot histórico.

F. Versionado optimistic

pos_products.version + pos_sales.version (integer) para concurrent updates. UPDATE ... WHERE version=N → si falla, retry.
